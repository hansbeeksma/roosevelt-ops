# ADR-010: Fastify voor API Layer

**Status:** Accepted
**Date:** 2026-02-26
**Deciders:** Roosevelt Digital Team

## Context

Roosevelt OPS heeft een dedicated API layer nodig, los van de Next.js front-end, voor:

1. **Agent-facing endpoints:** LLM-agents roepen de API aan voor data-operaties (Plane issue status, time entries, etc.)
2. **Externe integraties:** Webhooks van Twenty CRM, Inngest callbacks, Exact Online webhooks
3. **Client portal API:** Data-endpoints voor de client portal die niet via Next.js server actions gaan

Vereisten:
- Hoge performance (low latency voor agent workflows)
- TypeScript-native
- Goede plugin/module architectuur voor organische groei
- OpenAPI spec generatie voor agent tool definitions
- Eenvoudig te deployen naast de Next.js app

## Decision

We gebruiken **Fastify** als API framework met TypeScript.

Fastify's kernarchitectuur is gebaseerd op plugins — elke module registreert zijn eigen routes, hooks, en decorators. Dit past goed bij onze modular monolith aanpak (ADR-005).

**Configuratie:**
- `@fastify/swagger` voor automatische OpenAPI spec generatie
- Plugin-gebaseerde module structuur: `crm.plugin.ts`, `time.plugin.ts`, etc.
- `fastify-type-provider-zod` voor runtime request/response validatie
- Draait naast Next.js op een aparte port (of als separate Vercel function)

**Rationale:**
- Performance: Fastify is consistent sneller dan Express in benchmarks (~2x)
- Schema-first: built-in JSON schema validatie is sneller dan middleware-based validatie
- Plugin systeem: encapsulatie van scoping, hooks, en decorators per module
- OpenAPI generatie: essentieel voor LLM agent tool definitions
- TypeScript: volledige type-inferentie via type providers

## Consequences

### Positief
- Significante performance voordelen t.o.v. Express voor high-frequency agent calls
- OpenAPI spec dient als living documentation en agent tool schema
- Plugin architectuur dwingt module-grenzen af
- Actieve community en goede Next.js co-deployment support

### Negatief / Trade-offs
- Kleinere community dan Express (minder Stack Overflow antwoorden)
- Plugin incompatibiliteitsproblemen bij major upgrades
- Asynchroon plugin-registratie model kan verwarring geven
- Minder middleware-ecosystem dan Express

## Alternatieven overwogen

- **Express.js:** De industrie standaard, maar significant langzamer en geen built-in TypeScript support. Plugin/middleware model is minder gestructureerd. Afgewezen vanwege performance en gebrek aan moderne TypeScript-first design.
- **Hono:** Ultra-lichtgewicht, edge-native framework. Uitstekende performance maar kleiner ecosysteem. Minder plugin structuur voor grotere applicaties. Afgewezen vanwege minder volwassen plugin systeem voor onze use case.
- **tRPC:** Type-safe end-to-end RPC, maar koppelt front-end en back-end strak aan elkaar. Niet geschikt voor agent-facing endpoints of externe webhooks. Afgewezen vanwege beperkte use case (alleen TypeScript clients).
