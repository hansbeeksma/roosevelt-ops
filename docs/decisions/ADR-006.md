# ADR-006: NX Monorepo

**Status:** Accepted
**Date:** 2026-02-26
**Deciders:** Roosevelt Digital Team

## Context

Roosevelt OPS bestaat uit meerdere applicaties (web dashboard, client portal, Fastify API) die dezelfde types, utilities, en configuraties delen. Zonder gedeelde code-strategie leidt dit tot code-duplicatie en inconsistentie. Tegelijkertijd moeten builds snel blijven naarmate de codebase groeit.

We hadden een workspace-tool nodig die:
- Code sharing tussen apps mogelijk maakt via workspace-packages
- Intelligente caching biedt zodat alleen gewijzigde packages opnieuw worden gebuild
- Goed integreert met onze toolchain (TypeScript, Vitest, Next.js, Fastify)
- Schaalbaar is voor toekomstige toevoegingen

## Decision

We gebruiken een **NX workspace** met **pnpm** als package manager, met de volgende structuur:

```
roosevelt-ops/
├── apps/
│   ├── web/          # Next.js dashboard
│   ├── portal/       # Client portal (Next.js)
│   └── api/          # Fastify REST API
└── packages/
    ├── ui/           # Gedeelde shadcn/ui components
    ├── types/        # Gedeelde TypeScript types
    ├── config/       # Gedeelde configuraties (ESLint, TS, etc.)
    └── utils/        # Gedeelde utilities
```

**Rationale:**
- NX biedt computation caching: ongewijzigde packages worden niet opnieuw gebuild
- pnpm workspaces zijn efficiënter in disk gebruik dan npm/yarn (hard links)
- NX's affected commands (`nx affected:build`, `nx affected:test`) beperken CI tijd
- NX generator support voor consistente code-generatie
- Actief onderhouden met grote community

## Consequences

### Positief
- Significante vermindering van build-tijd via computation caching
- Één `pnpm install` voor alle apps en packages
- Gedeelde TypeScript types voorkomen type-inconsistentie tussen apps
- Gedeelde configuraties (Biome, TypeScript) zorgen voor consistente code style
- `nx graph` geeft inzicht in dependencies

### Negatief / Trade-offs
- NX heeft een leercurve voor nieuwe teamleden
- NX configuratie (`project.json`, `nx.json`) voegt complexiteit toe
- Remote caching (NX Cloud) vereist extra setup voor optimale CI performance
- Hogere initiële setup-tijd vergeleken met losse repos

## Alternatieven overwogen

- **Turborepo:** Vergelijkbare features, maar minder rijke generator/plugin ecosysteem. Minder volwassen tooling voor code generation. Afgewezen ten gunste van NX's rijkere feature set.
- **Separate repositories:** Eenvoudigste setup, maar maakt code sharing erg omslachtig. Vereist een private npm registry of `file://` dependencies. Afgewezen vanwege friction bij gedeelde type-updates.
- **Lerna:** Historisch de standaard voor monorepos, maar inmiddels ingehaald door NX en Turborepo. Afgewezen vanwege verouderde aanpak en beperkte caching-mogelijkheden.
