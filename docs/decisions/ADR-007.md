# ADR-007: Inngest voor Event-Driven Workflows

**Status:** Accepted
**Date:** 2026-02-26
**Deciders:** Roosevelt Digital Team

## Context

Roosevelt OPS heeft een reeks asynchrone workflows nodig die betrouwbaar moeten worden uitgevoerd, ook bij crashes, herstart, of piekbelasting:

- **Deal-pipeline:** `deal.won` event → Plane issue aanmaken → Slack notificatie → Twenty CRM update
- **Rapporten:** Wekelijkse time tracking rapporten, maandelijkse client summaries (cron jobs)
- **AI workflows:** Langlopende LLM-taken die meerdere stappen vereisen met retries
- **Webhooks:** Inkomende webhooks van Twenty CRM en Plane verwerken

Traditionele aanpak (direct API calls bij binnenkomst van events) is fragiel: als een stap faalt, verlies je de state. We hadden een durable execution engine nodig.

## Decision

We gebruiken **Inngest** voor alle event-driven workflows en scheduled jobs.

Inngest biedt durable functions: code die automatisch wordt hervat bij failures, zonder dat we de state handmatig hoeven op te slaan. De SDK integreert naadloos in onze Next.js en Fastify setup.

**Kerngebruik:**
```typescript
// Voorbeeld: deal.won pipeline
inngest.createFunction(
  { id: 'deal-won-pipeline' },
  { event: 'crm/deal.won' },
  async ({ event, step }) => {
    const issue = await step.run('create-plane-issue', () =>
      plane.createIssue(event.data)
    )
    await step.run('notify-slack', () =>
      slack.postMessage(event.data, issue)
    )
  }
)
```

**Rationale:**
- Durable execution: elke `step.run()` wordt gelogd en kan worden hervat
- Geen infrastructuur: Inngest beheert de queue en orchestratie
- Developer experience: functies zijn gewone TypeScript code
- Built-in retry logic, observability, en event history
- Lokale development via Inngest Dev Server (geen cloud account nodig)

## Consequences

### Positief
- Betrouwbare multi-step pipelines zonder handmatige state-management
- Automatische retries met exponential backoff
- Observability out-of-the-box (Inngest dashboard)
- Eenvoudig te testen lokaal via Dev Server
- Cron jobs en event-driven functions in dezelfde abstractie

### Negatief / Trade-offs
- Afhankelijkheid van Inngest cloud voor productie (vendor lock-in)
- Kosten schalen met function invocations (zie Inngest pricing)
- Inngest platform downtime beïnvloedt alle workflows
- Learning curve voor durable functions patronen

## Alternatieven overwogen

- **Bull/BullMQ:** Redis-gebaseerde job queue. Vereist Redis infra beheer. Geen durable execution (crashes = verloren jobs). Afgewezen vanwege operationele overhead en fragielheid bij crashes.
- **Temporal:** Krachtige durable workflow engine, maar zware setup (eigen cluster of Temporal Cloud). Overengineering voor onze schaal. Afgewezen.
- **AWS SQS/Lambda:** Krachtig maar brengt AWS vendor lock-in en significante IaC overhead. Niet passend bij onze Hetzner/Vercel-first strategie. Afgewezen.
- **Trigger.dev:** Vergelijkbaar met Inngest maar minder volwassen op moment van beslissing. Afgewezen ten gunste van Inngest's langere track record.
