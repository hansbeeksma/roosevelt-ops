# ADR-005: Modular Monolith Architecture

**Status:** Accepted
**Date:** 2026-02-26
**Deciders:** Roosevelt Digital Team

## Context

Roosevelt OPS is een intern operations platform voor een klein, gefocust team. We stonden voor de keuze welke architectuurstijl het beste past bij onze schaal en behoeften. Microservices zijn de industriestandaard voor grote systemen, maar brengen aanzienlijke operationele overhead mee die niet gerechtvaardigd is voor een intern team van < 10 personen. Een puur monolithische aanpak heeft echter ook nadelen op het gebied van onderhoudbaarheid naarmate het systeem groeit.

De kern vraag was: hoe houden we de codebase onderhoudbaar zonder overengineering?

## Decision

We kiezen voor een **modular monolith** architectuur bestaande uit twee primaire applicaties:

1. **Next.js (App Router)** — front-end + SSR, intern dashboard, client portal
2. **Fastify API** — REST API layer voor agent-facing endpoints en externe integraties

Elke applicatie is intern georganiseerd in duidelijk afgebakende modules (bijv. `crm/`, `time-tracking/`, `billing/`, `integrations/`). Modules communiceren via expliciete interfaces, niet via directe imports over module-grenzen.

**Rationale:**
- Operationele simpelheid: één deployable per applicatie, geen service mesh nodig
- Duidelijke module-grenzen vergemakkelijken latere extractie naar services als schaal dat vereist
- Gedeelde code (types, utils) zonder netwerk-overhead
- Klein team kan de volledige codebase overzien
- Snellere iteratie: geen inter-service versioning

## Consequences

### Positief
- Geen distributed systems complexity (geen service discovery, retries, circuit breakers)
- Eenvoudige lokale development setup
- Gedeelde transacties mogelijk (één database)
- Lager operationeel risico bij deployments
- Makkelijker te refactoren (alles in één codebase)

### Negatief / Trade-offs
- Schalen vereist horizontaal schalen van de hele applicatie (niet per module)
- Risico op het slijten van module-grenzen over tijd ("big ball of mud")
- Vereist architecturale discipline om grenzen te respecteren
- Kan een bottleneck worden als één module performance-intensief wordt

## Alternatieven overwogen

- **Microservices:** Elke module als aparte service met eigen deploy. Afgewezen vanwege onevenredige operationele overhead voor een intern platform op deze schaal.
- **Pure monolith:** Geen expliciete module-grenzen, alles in één codebase zonder structuur. Afgewezen vanwege onderhoudbaarheidsrisico's op lange termijn.
