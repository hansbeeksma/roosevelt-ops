# ADR-009: Clerk voor Authenticatie

**Status:** Accepted
**Date:** 2026-02-26
**Deciders:** Roosevelt Digital Team

## Context

Roosevelt OPS heeft twee verschillende gebruikersgroepen met verschillende authenticatie-behoeften:

1. **Medewerkers (intern):** Toegang via Google Workspace SSO. Moeten snel kunnen inloggen zonder extra wachtwoorden. RBAC op basis van rol (admin, developer, designer).
2. **Clients (extern):** Toegang tot client portal. Geen Google Workspace. Magic links zijn ideaal: geen wachtwoordbeheer voor incidentele gebruikers.

Vereisten:
- Multi-tenant: medewerkers en clients zijn aparte "audiences" met aparte rechten
- Google Workspace SSO voor het interne team
- Magic links voor client portal
- RBAC: rollen bepalen wat gebruikers kunnen zien en doen
- B2B SSO: toekomstige clients kunnen hun eigen SSO koppelen
- Next.js App Router compatibel (server components, middleware)

## Decision

We gebruiken **Clerk** als authenticatie provider.

Clerk is een managed auth service die de volledige auth stack afhandelt: sessies, tokens, gebruikersbeheer, en organisaties. De Next.js integratie is first-class.

**Configuratie:**
- Google Workspace SSO voor medewerkers via Clerk Organizations
- Magic link authenticatie voor client portal users
- `clerkMiddleware()` beschermt routes op basis van rol
- Clerk webhooks synchroniseren gebruikers naar onze database

**Rationale:**
- Zero-maintenance auth infrastructure: geen eigen JWT rotatie, sessie-opslag, of token validatie
- Google Workspace SSO out-of-the-box
- Organizations feature voor multi-tenancy (medewerkers in org, clients apart)
- Uitstekende Next.js App Router ondersteuning (server components, middleware)
- B2B SSO beschikbaar voor toekomstige enterprise clients

## Consequences

### Positief
- Geen auth-infrastructuur te onderhouden
- Veilig by default: Clerk handelt security best practices af
- Snelle implementatie: days, niet weken
- Rijke admin UI voor gebruikersbeheer
- Webhook-gedreven sync naar onze database

### Negatief / Trade-offs
- Vendor lock-in: migreren van Clerk is een significante operatie
- Kosten schalen met MAU (Monthly Active Users)
- Gevoelige gebruikersdata (emails, sessies) bij externe provider
- Customisatie-beperkingen voor edge cases

## Alternatieven overwogen

- **Supabase Auth:** Goed geïntegreerd met onze Supabase database, maar beperktere SSO/organization features. Google Workspace SSO vereist meer handmatige configuratie. Afgewezen vanwege minder rijke multi-tenant ondersteuning.
- **Auth0:** Volwassen platform, maar significant duurder en complexer te configureren. Overkill voor onze schaal. Afgewezen.
- **NextAuth.js (Auth.js):** Open-source, geen vendor lock-in, maar vereist zelf onderhoud van sessies, adapter, database schema. Meer flexibel maar ook meer werk. Afgewezen vanwege maintenance overhead voor een niet-differentiërende feature.
